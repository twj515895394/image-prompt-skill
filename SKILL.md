# Image Prompt Skill

## 使用目标

把用户的创作意图转成可直接投喂生图模型的完整 Prompt，而不是只给灵感词或风格清单。

默认根据任务需要产出：

- 可直接使用的主 Prompt
- 必要时的备选 Prompt
- 与任务直接相关的限制项
- 资产类任务需要的固定项、可变项或输出包结构

## 适用范围

支持以下任务：

- 成片生图
- 整体图生图改写
- 局部图片编辑
- 参考图反推 Prompt
- 角色资产
- 场景资产
- 分镜资产
- 视频首帧、尾帧和关键帧

不负责：

- 视频时间轴拆解
- 以运镜和时序为核心的完整视频 Prompt
- 视频下载、抽帧或文件解析

## 核心原则

- 先给可直接使用的结果，再补必要解释。
- 不把用户拉去填长表；能从文字、图片和上下文推断的内容直接推断。
- 交互模式遵循 `grill-me`：每次只问一个最关键的问题。
- 不用“高级感、电影感、氛围感”代替可执行的镜头、光影、色彩、材质和情绪描述。
- Reference 可以很多，但每次只读取当前任务真正需要的一条或几条路线。
- 同一知识只保留一个正文真源；不同索引可以指向它，但禁止复制正文。
- 不向用户暴露内部目录、维护路径和知识迁移过程。

## 第一步：判断任务

优先判断当前最接近哪一种任务，只读取一份任务 Playbook。

### 1. 成片生图 `finished-image`

生成一张高完成度结果图。产品图、海报、封面、写真、概念图等属于它的输出形态。

### 2. 整体图生图 `image-to-image`

保留部分参考锚点，对构图、场景、服装、光影、色调或风格进行整体改写。

### 3. 局部图片编辑 `image-editing`

只修改指定对象、区域、文字、动作或细节，并明确其他内容保持不变。

### 4. 参考图反推 Prompt `prompt-reverse-engineering`

根据用户意图选择：

- 复现型：尽量还原原图
- 结构型：提炼可复用视觉公式和模板
- 改造型：继承原图部分信息并生成新目标 Prompt

### 5. 角色资产 `character-assets`

生成三视图、多角度头像、表情板、换装板、手部、鞋履和角色主视觉等可复用资产。

### 6. 场景资产 `scene-assets`

生成场景总览、空景、站位图、道具关系、空间动线和镜头入口等资产。

### 7. 分镜资产 `storyboard-assets`

生成逐镜关键帧、动作拆解、故事板总板和控制型分镜资产。

### 8. 视频参考帧 `video-reference-frames`

生成图生视频使用的首帧、尾帧和关键过渡帧。

不要把角色、场景、分镜和参考帧统称为同一种资产图。

## 第二步：判断输入

输入层始终三选一：

- 只有文字：读取 `references/inputs/text-input-expansion.md`
- 单张参考图：读取 `references/inputs/single-image-reference.md`
- 多张参考图：读取 `references/inputs/multi-image-reference.md`

多图输入时不要平均融合。必须先判断每张图的职责，以及人物、服装、场景、构图、光影和风格分别由哪张图提供。

## 第三步：判断模式

### 快速模式

适用于用户目标已经明确，或明确要求直接出结果。

允许主动补全：

- 景别、机位、焦段
- 光源方向和色调结构
- 材质细节
- 画面层次
- 与任务相关的限制项

不得擅自锁死：

- 题材和世界观
- 明显改变方向的审美大类
- 关键人物关系
- 用户没有暗示的强剧情设定

即使快速模式也必须追问的情况：

- 缺少主体，无法判断画面围绕谁或什么
- 多图互相冲突，无法分配参考职责
- 同时存在互斥目标
- 角色资产缺少最基本的一致性锚点

### 交互模式

适用于用户明确要求共创、找方向或逐步讨论。

规则：

- 每次只问一个问题
- 只问当前最影响结果的变量
- 能推断则不追问
- 允许指出模糊、冲突或不可同时满足的要求
- 用户喊停，或再问只剩微调时立即收口

## Reference 读取流程

1. 读取 `references/index.md`。
2. 固定读取：
   - `1` 份 input Reference
   - `1` 份 task Reference
3. 按需读取：
   - `0-2` 份 controls
   - `0-2` 份 libraries
   - `0-1` 份 style
   - `0-1` 份 diagnostic
4. 分类索引只负责导航，不计入业务 Reference 数量。
5. 不为了凑满额度而读取资料。
6. 同一文件被多个入口命中时只读取一次。
7. 只有用户反馈效果不好、参考冲突或问题跨多个维度时才读取 diagnostics。

## Reference 职责

- `inputs/`：如何理解文字、单图和多图输入
- `tasks/`：任务目标、执行流程和输出结构
- `controls/`：如何判断、选择、协调和控制
- `libraries/`：具体选择器、详细资料和知识库
- `styles/`：某种视觉风格如何实现
- `diagnostics/`：跨任务、跨维度综合诊断

## 信息优先级

发生冲突时按以下顺序处理：

1. 用户当前明确要求
2. 用户明确指定必须保留或必须修改的内容
3. 当前任务 Playbook
4. 当前输入 Reference
5. 已加载的控制规则
6. 已加载的资料库与风格资料
7. 自动补全项

局部编辑任务中，“其他内容保持不变”属于高优先级约束。

## Prompt 组装原则

- 先确定主体与任务目标，再写风格。
- 先写空间、动作和观看关系，再写装饰细节。
- 镜头、光影、色彩、材质和情绪必须互相支持。
- 负向限制只针对当前最危险的失败模式，不做无差别堆叠。
- 参考图任务必须明确：保留什么、修改什么、允许什么变化、禁止什么出现。
- 默认使用中文完整描述，可保留必要英文行业术语和关键词。

## 输出前自查

- 主体是否唯一清楚
- 场景是否可视觉化
- 动作、重心和接触关系是否成立
- 镜头与光影是否互相支持
- 参考图职责是否清楚
- 固定项与可变项是否混淆
- 限制项是否与任务直接相关
- 是否出现互相冲突的要求

## 输出要求

普通任务至少包含：

- 画面或任务概述
- 最终 Prompt
- 任务相关限制项

需要时补充：

- 备选 Prompt
- 保留项 / 修改项 / 禁止变化项
- 固定项 / 可变项
- 推荐资产输出顺序
- 已自动补全项
- 方向决策摘要

用户要求完整结构化交付时，按需读取当前任务对应的 `assets/templates/` 模板；普通短任务不强制读取模板。

## 严禁事项

- 不把维护说明写进对外输出
- 不在未必要时解释内部推理
- 不一次性加载整个 Reference 目录
- 不复制同一知识形成多个正文版本
- 不先堆风格词再补主体
- 不用空泛质量词替代可执行视觉变量
